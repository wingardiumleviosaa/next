在容器裡 curl Kubernetes API server

Publish on: 2022-03-31 Classify at: Kubernetes
Words: 475 Read:≈ 1mins

連接 k8s 的 api-server 有三種方式：

  1. Kubernetes Node 通過 kubectl proxy 中轉連接
  2. 通過授權驗證直接連接，例如 kubectl 和各種 client 就是這種情況
  3. 容器內部通過 ServiceAccount 連接

本文以第三種情況作範例。

Kubernetes API Server

在 Kubernetes 集群被創建時，預設會在 default namespace 中創建 kubernetes 的服務，用於訪問 Kubernetes apiserver。因此，Pod 之間可以直接使用 kubernetes.default.svc 主機名來查詢 API server。

Service Account

ServiceAccount 是給執行在 Pod 的程式使用的身份認證，給 Pod 容器的程式訪問 API Server 時使用；ServiceAccount 僅侷限它所在的 namespace，每個 namespace 建立時都會自動建立一個 default service account；建立 Pod 時，如果沒有指定 Service Account，Pod 則會使用 default Service Account。

Service Account Secret

SA 對應的 Secret 會自動掛載到 Pod 的 /var/run/secrets/kubernetes.io/serviceaccount/ 目錄中(包含 token、ca.crt、namespace)。

創建 Role & Role Binding

如果直接使用預設的 sa 訪問 api server 會遇到權限不足的問題

此時需要建立角色開放存取 api 指定路徑的權限並綁定角色到 SA 上

 1# role&binding.yaml
 2---
 3apiVersion: rbac.authorization.k8s.io/v1
 4kind: Role
 5metadata:
 6  name: default-role
 7  namespace: converg-it
 8rules:
 9  - apiGroups: [""]
10    resources:
11      - pods
12      - pods/log
13    verbs:
14      - get
15      - list
16---
17apiVersion: rbac.authorization.k8s.io/v1
18kind: RoleBinding
19metadata:
20  name: default-roldbinding
21  namespace: converg-it
22subjects:
23  - kind: ServiceAccount
24    name: default
25roleRef:
26  kind: Role
27  name: default-role
28  apiGroup: rbac.authorization.k8s.io

1kubectl -n converg-it apply -f role&binding.yaml
2role.rbac.authorization.k8s.io/default-role created
3rolebinding.rbac.authorization.k8s.io/default-roldbinding created

curl API

進入容器環境

1kubectl exec -it -n converg-it converg-it-adapter-oracle-8575f54dc6-7lnv6 -- bash

get target API

1curl --cacert /var/run/secrets/kubernetes.io/serviceaccount/ca.crt --header "Authorization: Bearer $(cat /var/run/secrets/kubernetes.io/serviceaccount/token)" -X GET https://kubernetes.default.svc/api/v1/namespaces/converg-it/pods/converg-it-adapter-oracle-8575f54dc6-7lnv6/log?sinceSeconds=300
